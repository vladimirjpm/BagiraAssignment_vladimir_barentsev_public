Scenario Builder — Шпаргалка для защиты

.NET 10 Web API · Clean Architecture · In-memory → DB-ready · React + Vite · Docker Compose · xUnit

Статус: реализованы и проверены Phase 1 (сценарии) и Phase 2 (сущности) — CRUD, server-side фильтр + сортировка, валидация, единый ErrorResponse, no-orphan, реальный entityCount, связка фронт↔бэк через vite-proxy. Осталось: глобальный поиск (/api/search), Docker (compose/nginx), мелкий полиш сообщения при невалидном enum, опц. тесты. Блоки в плане ещё не сделаны.

Масштабирование — путь in-memory → production

1

In-memory → реальная БД (главный заготовленный шаг)

Сейчас данные лежат в InMemoryStore (Singleton, два ConcurrentDictionary), а InMemory*Repository читают из него. Переход на БД = написать EfCoreScenarioRepository : IScenarioRepository + DbContext, поменять регистрацию в AddInfrastructure(). Domain, Application и контроллеры не меняются (DIP + OCP). Это и есть смысл «DB-ready».

2

Пагинация и индексы

Фильтрация уже server-side. При росте данных добавляем ?page=&pageSize= и переносим фильтр/сортировку в SQL (IQueryable), чтобы не тянуть всю таблицу в память. На БД — индексы по полям фильтрации (name, type, taskForce).

3

Горизонтальное масштабирование API

После выноса состояния в БД API становится stateless — можно поднять N реплик за load balancer. In-memory вариант этого не позволяет (состояние в процессе), поэтому БД — предусловие для масштабирования вширь.

Вопросы на защите — готовые ответы

Почему Clean Architecture, а не один проект?
Разделение ответственности + тестируемость + замена хранилища без боли
Domain в центре, зависимости направлены внутрь. БД, веб и фреймворк — детали на краю. Бизнес-правила не зависят от ASP.NET и можно юнит-тестировать без поднятия сервера.
Почему сначала in-memory, а не сразу БД?
24-часовой таймбокс — приоритет на архитектуру и рабочий docker-compose
Репозитории за интерфейсами, поэтому БД — это «stage 2», добавляется без переписывания. Показываю готовность к БД, не тратя время на миграции в рамках теста.
Зачем DTO, если есть доменные модели?
Контракт API отделён от внутренней модели
Можно менять домен, не ломая клиента, и наоборот. Валидация и форма ответа живут в DTO; доменная модель остаётся чистой. Плюс не отдаём наружу лишние поля.
Где фильтрация — на клиенте или сервере?
На сервере — это в Acceptance Criteria
Фильтр по name/description для сценариев и по type/taskForce для сущностей делается в репозитории. Готово к переносу в SQL без смены контракта.
Как обрабатываете ошибки?
400 на невалидный ввод, 404 на отсутствие, единый ErrorResponse
Форма: { message, status, errors? } — поле message критично, его читает entityService. Две точки: доменные исключения ловит ExceptionHandlingMiddleware (NotFoundException→404, DomainException→400, прочее→500); авто-400 модель-валидации переопределён через InvalidModelStateResponseFactory в ту же форму (а не дефолтный ProblemDetails с title).
Как гарантируете «нет сирот»?
Создание Entity проверяет существование Scenario → иначе 404
Сущность всегда привязана к сценарию через scenarioId в маршруте; репозиторий/сервис отклоняет создание для несуществующего сценария.
Как запускается всё вместе?
docker-compose up — backend + frontend
Multi-stage Dockerfile для фронта (build → serve через nginx). API URL фронта и CORS-origins бэка — через переменные окружения, не хардкод.
Использовал AI при разработке?
Да — для обсуждения архитектуры, trade-offs и ускорения рутинного кода
Весь код просмотрен и адаптирован под задание. Решения (слои, контракт, in-memory→DB) принимал сам; AI — инструмент, как IntelliSense, а не автор архитектуры.

Каверзные вопросы — то что реально спрашивают

ЛОВУШКИ НА ЗАЩИТЕ

Почему путь сущностей такой странный — /api/entities/scenarios/{id}/entities?
Так задан сервисами фронта — контракт фиксирован клиентом
Я не проектировал URL с нуля: фронтенд-скелет уже зовёт именно этот путь (entityService.ts). Бэкенд обязан совпасть, иначе 404. RESTful-альтернативу (/api/scenarios/{id}/entities) мог бы предложить, но менять контракт = ломать данный фронт.
In-memory + несколько потоков — нет ли гонок?
Да — состояние в InMemoryStore на двух ConcurrentDictionary<Guid, T>
ASP.NET обрабатывает запросы параллельно, поэтому обычный Dictionary/List дал бы race condition. Id генерируются как Guid.NewGuid() в доменной фабрике, запись по ключу — атомарный индексатор словаря.
Singleton или Scoped для репозитория?
Хранилище — Singleton, репозитории — Scoped (читают из него)
Состояние должно переживать запросы → InMemoryStore зарегистрирован как Singleton. Сами репозитории Scoped — так при переходе на EF Core ничего не меняется (там DbContext и репозиторий обязаны быть Scoped). Singleton-репозиторий, держащий Scoped DbContext, был бы Captive Dependency.
Валидация enum: что если придёт type: "Banana"?
Отклоняем на этапе модель-байндинга → 400, не маппим в домен
Enum'ы сериализуются строками (JsonStringEnumConverter); неизвестное значение не парсится → 400 (а не 500), в едином ErrorResponse. Latitude/Longitude — [Range] (-90..90 / -180..180), плюс backstop в доменной фабрике. Полиш: дефолтное сообщение байндера техническое — стоит заменить на дружелюбное.
CORS или proxy? Почему относительные пути 404 в dev?
Гибрид: same-origin по умолчанию, CORS — подстраховка
Сервисы фронта зовут относительные /api/.... В dev на :5173 это ушло бы в сам Vite (404), CORS тут не спасёт — запрос даже не cross-origin. Решение: vite-proxy /api → :5000 в dev и nginx в Docker → браузер видит same-origin, CORS не нужен. База берётся из VITE_API_BASE_URL ?? ''; если выставить cross-origin URL — включается CORS-политика из ENV бэка (Cors:AllowedOrigins).
Где сортировка и как защищена от инъекции по sortBy?
sortBy — из белого списка полей, не прямая подстановка
Параметр sortBy мапится на разрешённые поля через switch/словарь; неизвестное значение → дефолт или 400. Это исключает «сортировку по произвольному выражению» и будущую SQL-инъекцию при переходе на БД.
Что если Docker не успел — это провал?
Нет — задание разрешает письменное описание контейнеризации
В задании прямо сказано: если Docker занимает больше времени, допустимо описать в README, как бы контейнеризировал. Но Docker помечен критичным, поэтому цель — рабочий docker-compose up, а текст — запасной вариант.

Program.cs — реальная конфигурация

// Строковые enum'ы — контракт фронта ("Soldier", "Friendly") builder.Services.AddControllers().AddJsonOptions(o => { o.JsonSerializerOptions.PropertyNamingPolicy = JsonNamingPolicy.CamelCase; o.JsonSerializerOptions.Converters.Add(new JsonStringEnumConverter()); }); // Единый формат 400: авто-валидацию заворачиваем в ErrorResponse builder.Services.Configure<ApiBehaviorOptions>(o => o.InvalidModelStateResponseFactory = ctx => new BadRequestObjectResult( new ErrorResponse("Validation failed.", 400, /* errors dict */))); // CORS origins из ENV (Cors__AllowedOrigins__0) var origins = builder.Configuration.GetSection("Cors:AllowedOrigins").Get<string[]>() ?? ["http://localhost:5173"]; builder.Services.AddCors(o => o.AddPolicy("frontend", p => p.WithOrigins(origins).AllowAnyHeader().AllowAnyMethod())); // DI: store=Singleton, репозитории=Scoped (см. AddInfrastructure) builder.Services.AddInfrastructure(); builder.Services.AddScoped<IScenarioService, ScenarioService>(); builder.Services.AddScoped<IEntityService, EntityService>(); // Pipeline: middleware ошибок первым; без UseHttpsRedirection (Docker) app.UseMiddleware<ExceptionHandlingMiddleware>(); app.UseCors("frontend"); app.MapControllers();
